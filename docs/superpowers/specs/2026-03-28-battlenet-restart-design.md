# 战网实例约束的炉石自动重启设计

日期：2026-03-28

## 1. 背景

当前仓库里存在两套“启动炉石”的机制：

1. `BotMain/BotService.cs` 的自动重启链路会在匹配超时或游戏闪退后，直接调用 `Process.Start(Hearthstone.exe)` 拉起炉石。
2. `BotMain/AccountController.cs` 已经实现了另一套机制：先关闭当前炉石，再通过已绑定的战网窗口 PID，调用 `BattleNetWindowManager.LaunchHearthstoneFrom(...)` 点击战网“开始游戏”按钮来启动炉石。

这两套机制并存会带来两个问题：

- 自动重启与中控切号的行为不一致，导致排查问题时很难判断究竟是哪条启动链路生效。
- 自动重启直接从游戏路径启动炉石，不符合当前实际约束。目标行为应当是：炉石重启必须从战网实例发起，而不是直接执行 `Hearthstone.exe`。

本次设计的目标是统一这两条链路，让“脚本自动重启炉石”和“中控切号启动炉石”都建立在同一个战网实例启动能力之上。

## 2. 需求与约束

### 2.1 必须满足的需求

- 自动重启炉石时，不能再直接执行 `Hearthstone.exe`。
- 自动重启只能复用“当前账号已绑定的战网窗口 PID”。
- 战网 PID 必须是当前运行会话显式注入给 `BotService` 的上下文，而不是运行时临时扫描任意战网窗口。
- 匹配超时重启和游戏闪退重启都必须遵循同一条战网启动路径。

### 2.2 用户已确认的约束

- 自动重启的战网来源选择 `A`：只复用已绑定的战网窗口 PID。
- 如果当前运行的 bot 没有绑定战网窗口 PID，则自动重启失败，直接停止脚本并明确报错。
- 如果当前账号在自动重启时发现绑定的战网 PID 已失效，或者从战网点击启动炉石超时，则直接停止整个队列并明确报错。

### 2.3 非目标

- 不在本次设计里新增“普通单号模式的战网配置界面”。
- 不在本次设计里支持“自动从系统里选择唯一一个战网窗口”。
- 不保留 `Process.Start(Hearthstone.exe)` 作为自动重启的兜底路径。

## 3. 方案比较

### 3.1 方案 A：`BotService` 直接依赖 `AccountController`

让 `BotService` 在重启时直接读取当前账号对象，从 `CurrentAccount.BattleNetProcessId` 取到战网 PID 后重启。

优点：

- 表面上改动最少。

缺点：

- `BotService` 反向依赖中控层，职责关系混乱。
- `BotService` 会知道太多“队列账号”概念，不利于单号模式和后续维护。

结论：

不采用。

### 3.2 方案 B：把战网实例上下文下沉为 `BotService` 的运行时配置

保留 `BattleNetWindowManager` 作为底层启动器，`BotService` 新增一份“当前会话绑定的战网 PID”上下文。中控切换账号时，将当前账号绑定的战网 PID 同步给 `BotService`。自动重启时，`BotService` 只依赖这份上下文完成启动。

优点：

- 依赖方向正确。
- `BotService` 无需知道中控队列细节。
- 自动重启与中控切号可以复用同一套战网启动能力。

缺点：

- 需要补一层 `MainViewModel` / `AccountController` 到 `BotService` 的上下文同步。

结论：

采用此方案。

### 3.3 方案 C：把自动重启整体上移到 `AccountController`

让 `BotService` 只负责发现断线，真正的自动重启全部由 `AccountController` 驱动。

优点：

- 中控模式下的行为集中在中控层。

缺点：

- 单号模式会失去现有自动重启能力。
- 需要重构现有 `TryReconnectLoop` 以及相关重连流程，改动面过大。

结论：

本次不采用。

## 4. 总体设计

### 4.1 职责划分

#### `BattleNetWindowManager`

负责底层“通过指定战网 PID 启动炉石”的能力：

- 校验 PID 对应窗口是否存在。
- 前置窗口。
- 计算按钮坐标并点击“开始游戏”。
- 等待 `Hearthstone` 进程出现。

它只负责“怎么从一个已知战网实例启动炉石”，不负责任何业务判断。

#### `BotService`

负责自动重启与重连主链路：

- 保存当前会话绑定的战网 PID。
- 在匹配超时或游戏闪退时决定是否发起自动重启。
- 调用战网启动器拉起炉石。
- 继续复用现有 `TryReconnectLoop()` 等待 payload 重连。
- 当战网上下文缺失或战网启动失败时，产出明确的不可恢复失败结果。

`BotService` 不直接依赖 `AccountController`，只依赖“当前战网 PID”这份运行时上下文。

#### `MainViewModel` / `AccountController`

负责产品层编排：

- 在切换到某个账号时，把该账号的 `BattleNetProcessId` 注入给 `BotService`。
- 在自动重启出现不可恢复失败时，停止 bot，并进一步停止整个队列。
- 把失败原因转成清晰的 UI 状态和日志。

### 4.2 统一启动原则

本次设计落地后，炉石启动行为统一为两条：

1. **中控切号启动**
   由 `AccountController.SwitchToAccount()` 主动调用 `BattleNetWindowManager.LaunchHearthstoneFrom(...)`。

2. **脚本自动重启启动**
   由 `BotService` 在重启链路里调用同一套战网启动能力。

这两条链路共享同一套底层战网启动能力，不再存在“中控走战网、自动重启走游戏路径”的分叉。

## 5. 数据流设计

### 5.1 运行前注入战网上下文

当 `AccountController` 切换到一个新账号时，当前流程除了：

- 停 bot
- 关闭当前炉石
- 从目标战网窗口启动炉石
- 应用账号配置
- 启动 bot

还要新增一步：

- 将 `account.BattleNetProcessId` 注入到 `BotService`

这一步在语义上表示：

`从现在开始，这个 bot 会话只能使用这个战网实例进行自动重启。`

### 5.2 匹配超时重启链路

当前 `RestartHearthstone()` 会：

- 杀掉所有 `Hearthstone` 进程
- 重置 pipe / prepared 状态
- 直接 `Process.Start(Hearthstone.exe)`

新设计改为：

1. 杀掉所有 `Hearthstone` 进程
2. 重置 pipe / prepared / decks 状态
3. 读取当前保存的战网 PID
4. 校验该 PID 是否有效
5. 调用战网启动器从指定 PID 启动炉石
6. 若启动成功，记录日志并等待 `TryReconnectLoop()` 后续连接成功
7. 若启动失败，立即进入不可恢复失败分支

### 5.3 闪退重启链路

`TryReconnectLoop()` 目前会在发现 `Hearthstone` 进程已经消失时尝试重新启动游戏。

新设计里，这一步不再解析 `Hearthstone.exe` 路径，也不再尝试直启，而是：

1. 发现游戏进程消失
2. 校验当前保存的战网 PID
3. 从该战网实例启动炉石
4. 继续复用现有 15 秒周期的无限重连逻辑

这意味着：

- 匹配超时重启和闪退重启的“启动炉石”动作完全统一
- 差别只剩下触发原因不同，日志原因不同

### 5.4 失败上报链路

`BotService` 需要新增一种明确的上报能力，用于向上层说明：

- 自动重启失败了
- 失败原因是什么
- 这是不可恢复的，不应继续重试

建议对上层暴露一类语义清晰的事件/回调，例如：

- `OnRestartFailed(string reason)`

该事件只在不可恢复失败时触发，不用于普通的“仍在重连中”状态。

上层收到这个事件后的固定行为：

1. 记录日志
2. 停止 bot
3. 如果当前处于账号队列运行中，则停止整个队列
4. 将状态文案切换为明确失败态

## 6. 不可恢复失败定义

以下情况全部视为不可恢复失败，不允许降级为本地直启炉石：

### 6.1 未绑定战网实例

当前运行会话没有战网 PID。

处理：

- 触发自动重启失败事件
- 日志写明“未绑定战网实例”
- 停止脚本

### 6.2 战网实例已失效

保存的 PID 已不存在，或者 `BattleNetWindowManager.IsProcessAlive(pid)` 返回 false。

处理：

- 触发自动重启失败事件
- 日志写明“战网实例已退出”
- 停止脚本
- 若处于中控队列，则停止整个队列

### 6.3 战网窗口无法操作

包括：

- 找不到窗口句柄
- 无法前置窗口
- 无法获取窗口坐标

处理：

- 触发自动重启失败事件
- 日志写明具体失败阶段
- 停止脚本 / 队列

### 6.4 战网点击启动后超时

点击“开始游戏”后，在既定超时窗口内仍未检测到 `Hearthstone` 进程出现。

处理：

- 触发自动重启失败事件
- 日志写明“从战网启动炉石超时”
- 停止脚本 / 队列

## 7. 单号模式与中控模式的行为

### 7.1 中控模式

中控模式是本次设计的主要工作模式：

- 每个账号都已经显式绑定战网 PID
- 切号时会把 PID 注入给 `BotService`
- 自动重启失败时，整队停止，不跳到下一个账号

这符合用户已确认的约束：失败时宁可整队停机，也不要在错误状态下继续切号。

### 7.2 普通单号模式

按照本次约束，普通单号模式如果没有战网 PID 上下文，则不再具备自动重启能力。

具体表现为：

- 正常运行不受影响
- 一旦触发匹配超时重启或闪退重启
- `BotService` 会判定“未绑定战网实例”
- 自动重启失败，脚本停止

这是本次设计的有意结果，而不是副作用。因为当前用户要求已经明确：

`没有绑定战网窗口 PID 时，不能自动重启，也不能退回到直接启动 Hearthstone.exe。`

## 8. 日志与状态设计

### 8.1 日志原则

日志必须能直接区分：

- 重启触发原因
- 使用了哪个战网 PID
- 失败发生在哪个阶段

建议日志样式包括：

- `[Restart] 匹配超时重启：使用战网实例 PID=1234 启动炉石`
- `[Restart] 游戏闪退：使用战网实例 PID=1234 启动炉石`
- `[Restart] 自动重启失败：未绑定战网实例`
- `[Restart] 自动重启失败：战网实例已退出 PID=1234`
- `[Restart] 自动重启失败：无法前置战网窗口 PID=1234`
- `[Restart] 自动重启失败：从战网启动炉石超时 PID=1234`
- `[中控] 因自动重启失败已停止队列`

### 8.2 状态原则

当前 `Reconnecting` 只适合“仍有机会恢复”的场景。

因此：

- 成功发起战网启动并等待 payload 重连时，可以继续显示 `Reconnecting`
- 一旦进入不可恢复失败，状态必须切换为明确失败态，而不是继续保留 `Reconnecting`

失败状态文案应当包含明确原因，避免 UI 看起来像“还会自动恢复”。

## 9. 涉及文件

### 9.1 `BotMain/BotService.cs`

需要改动：

- 新增或调整“当前绑定战网 PID”的运行时字段及设置方法
- 改造 `RestartHearthstone()`，删除直接 `Process.Start(Hearthstone.exe)` 的路径
- 改造 `TryReconnectLoop()` 中“进程消失后自动启动”的路径，改为走战网实例
- 新增自动重启失败事件/回调

### 9.2 `BotMain/AccountController.cs`

需要改动：

- 在切换账号时把 `BattleNetProcessId` 同步给 `BotService`
- 订阅或响应自动重启失败后的上层停机逻辑

### 9.3 `BotMain/MainViewModel.cs`

需要改动：

- 作为 `BotService` 与 `AccountController` 的桥接层，承接自动重启失败事件
- 在失败时更新 UI 状态，并驱动中控停止队列

### 9.4 `BotMain/BattleNetWindowManager.cs`

可能需要小幅改动：

- 补充更细粒度的失败日志返回
- 视实现需要，拆分出更明确的失败原因，供 `BotService` 记录

## 10. 测试策略

### 10.1 核心回归场景

1. **匹配超时重启，PID 有效**
   - 当前账号已绑定有效战网 PID
   - 匹配超时触发 `RestartHearthstone()`
   - 预期：炉石从指定战网实例启动，随后成功重连

2. **对局中闪退重启，PID 有效**
   - 对局中强制关闭 `Hearthstone.exe`
   - 预期：`TryReconnectLoop()` 检测到进程消失，从指定战网实例启动炉石，并最终重连成功

3. **未绑定战网 PID**
   - 当前 bot 会话未注入战网 PID
   - 触发匹配超时或闪退重连
   - 预期：日志明确报“未绑定战网实例”，脚本停止

4. **战网 PID 已退出**
   - 当前账号保存的战网 PID 对应进程已关闭
   - 触发自动重启
   - 预期：自动重启失败，脚本停止；如果在中控队列里，则整队停止

5. **战网启动超时**
   - 战网窗口仍存在，但点击开始后 90 秒内没有出现 `Hearthstone` 进程
   - 预期：自动重启失败，脚本 / 队列停止，日志保留超时原因

### 10.2 兼容性检查

还需确认以下兼容点没有被破坏：

- 中控切号仍能正常从战网实例启动炉石
- 原有 `TryReconnectLoop()` 的 payload 重连节奏保持不变
- 手动 `Stop` 时不会误触发自动重启失败事件

## 11. 验收标准

本次设计实现完成后，应满足以下验收标准：

- 自动重启不再直接调用 `Process.Start(Hearthstone.exe)` 启动炉石
- 自动重启只使用当前会话绑定的战网 PID
- 匹配超时重启与闪退重启都走同一套战网启动链路
- 没有战网 PID、PID 已失效、战网启动超时等场景都会明确失败并停机
- 中控队列在自动重启不可恢复失败时会停止整个队列，而不是切下一个账号
- UI 与日志能明确区分“仍在重连中”和“自动重启失败”

## 12. 风险与后续

### 12.1 已知风险

- 当前 `BattleNetWindowManager` 依赖固定按钮坐标点击“开始游戏”，如果战网 UI 布局变化，启动动作可能失效。
- 普通单号模式在未注入战网 PID 时会失去自动重启能力，这符合当前约束，但需要在日志和状态上足够明确，避免被误判为 bug。

### 12.2 后续可选工作

本次不纳入实现，但后续可以考虑：

- 为普通单号模式补一套显式的战网实例配置能力
- 让 `BattleNetWindowManager` 返回结构化失败码，而不是只返回 `true/false`
- 为战网窗口点击坐标提供校准或配置入口，降低 UI 偏移导致的失败率
