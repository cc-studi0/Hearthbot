# 多选卡组按局随机设计

**日期**: 2026-04-16  
**范围**: 主界面卡组选择、中控账号卡组配置、构筑开局选牌、运行时卡组上下文  
**目标**: 允许用户同时选择多个构筑卡组，并在每一局开始匹配前从已选卡组中随机选择一套进入对局，且本局后续留牌、发现、学习上下文都与实际选中的卡组一致

## 背景

当前脚本的卡组配置链路是单值模型：

- 主界面 `MainViewModel` 只保存一个 `DeckName`
- 中控 `AccountEntry` / `accounts.json` 也只保存一个 `DeckName`
- `BotService.SetRunConfiguration()` 与 `AccountController.ApplyAccountSettings()` 最终都把单个卡组名写进 `BotService._selectedDeck`
- 真正进入构筑匹配前，`BotService.AutoQueueConstructedByUi()` 再用 `_selectedDeck` 查 deck id 并执行 `SELECT_DECK`

这意味着如果只在 UI 上做“多选展示”，而不改运行态，脚本仍会把留牌、发现、学习数据归到旧的单一卡组上，行为会失真。

## 设计目标

### 1. 每局重新随机

随机粒度是“每一局开始前”，不是“每次点击 Start 后固定整轮运行”。

### 2. 主界面与中控一致

主界面单独运行和中控多账号运行都支持多选卡组，表达方式和随机规则保持一致。

### 3. 兼容旧配置

已有的 `DeckName` 单值配置必须继续可读，不要求用户手动迁移。

### 4. 运行态一致

一旦本局随机出了实际卡组，后续：

- 对局启动时选择的卡组
- `ResolveDeckContext()` 生成的 `DeckName` / `DeckSignature`
- 留牌/发现/学习记录里的卡组信息

都必须使用这局实际卡组，而不是“候选列表”或旧的单值配置。

## 方案选择

采用 **方案 A：保留现有 Deck 显示位，新增多选入口**。

### 选择原因

- 改动集中在配置表达和启动前决策，不必重做主窗口布局
- 可以保留旧的单选兼容路径，降低回归风险
- 中控编辑框也能复用同一套多选弹窗/选择逻辑

### 不采用的方案

- 不直接把主界面顶部 `Deck` 下拉改成大面积多选列表，避免牵动现有布局
- 不做复杂的自定义下拉标签面板，避免在 WPF 中引入额外交互和样式成本

## 总体方案

引入“候选卡组列表”和“本局实际卡组”两个概念。

### 候选卡组列表

由用户在主界面或中控中勾选多个卡组，作为随机池。

### 本局实际卡组

在每次准备开始构筑匹配前，从候选池中随机选择一个卡组；这一结果在本局生命周期内固定，直到下一局重新匹配前再次随机。

## 配置模型

### 主界面

在 `MainViewModel` 中新增多选卡组状态：

- `SelectedDeckNames`：当前勾选的候选卡组列表
- `DeckSelectionSummary`：用于 UI 展示的压缩文本，例如 `随机(3)` 或 `机械法 等3个`

保留已有的 `DeckName` 持久化兼容，但新增数组字段：

- `SelectedDeckNames`

读取设置时：

1. 如果新字段存在，优先按数组加载
2. 如果只有旧 `DeckName`，则自动迁移为单元素数组
3. 如果数组为空，则回退到 `(auto)`

### 中控账号

在 `AccountEntry` 中新增：

- `SelectedDeckNames`
- `DeckSummary`

持久化到 `accounts.json` 时新增数组字段：

- `SelectedDeckNames`

读取账号配置时：

1. 优先读数组
2. 否则回退到旧 `DeckName`

## UI 设计

### 主界面

保留当前 `Deck` 区域，但不再把下拉框作为唯一配置入口。

推荐表现：

- 主展示文本显示候选池摘要
- 保留刷新卡组按钮
- 新增 `多选` 按钮打开卡组勾选窗口

摘要规则：

- 0 个候选：显示 `(auto)`
- 1 个候选：显示该卡组名
- 2 个及以上：显示 `随机(n)`

### 中控编辑框

`AccountEditDialog` 中原有单选 `DeckCombo` 改为：

- 一个只读摘要显示
- 一个 `多选` 按钮

账号列表中的“卡组”列继续保留，但显示摘要文本而不是单一卡组名。

## 随机时机

随机必须发生在“每次准备进入构筑匹配”之前。

具体规则：

1. 主循环进入构筑自动排队路径时，如果当前不在对局内且即将执行 `GET_DECK_ID` / `SELECT_DECK`
2. 先解析当前候选卡组列表
3. 从候选池中随机选出本局卡组
4. 记录到运行态字段，例如 `_activeMatchDeck`
5. 用该值继续查 deck id、切卡组、写日志

下一局结算回到大厅并再次准备匹配时，重复上述流程，重新随机。

## 运行态一致性

新增运行态字段，将“配置候选池”和“本局实际卡组”分离。

建议语义：

- `_selectedDecks`：配置上的候选池
- `_selectedDeck` 或新字段 `_activeMatchDeck`：本局已经随机出的实际卡组

要求：

- `SetRunConfiguration()` 和 `SetDeckByName()` / 新增多选接口应设置候选池
- 只有在真正准备开局时，才确定本局实际卡组
- `ResolveDeckContext()` 优先使用本局实际卡组
- 若没有实际卡组但只有单个候选卡组，可以退化到单卡组逻辑

这样可以确保学习与对局行为归属正确。

## 随机策略

初版使用均匀随机，不引入权重、去重轮巡或“最近卡组避让”。

候选池处理规则：

- 忽略空值
- 去重
- 候选池只有 1 个时直接使用该卡组，不记录“随机失败”
- 候选池为空时回退 `(auto)` 逻辑

## 日志与可观察性

每局随机后输出明确日志，例如：

- `[DeckRandom] 候选=3 实际=星灵法 (Mage)`
- `[DeckRandom] 候选为空，回退到(auto)`

中控切号时也保留配置摘要，方便区分“账号配置的候选池”和“本局实际选中的卡组”。

## 兼容性策略

### 设置兼容

- 继续写旧 `DeckName`，值取“首个候选卡组”或 `(auto)`，用于老版本容错
- 新版本真实行为以 `SelectedDeckNames` 为准

### 中控兼容

- `accounts.json` 中保留 `DeckName`
- 新增 `SelectedDeckNames`
- 老账号文件无需手改即可继续使用

### 现有 API 兼容

保留现有单卡组接口，新增多卡组接口；旧调用方不需要一次性全部重写。

## 测试策略

### 单元测试

在 `BotCore.Tests` 为 `BotService` 增加以下覆盖：

1. 候选池为空时，随机结果回退 `(auto)`
2. 候选池只有一个卡组时，总是返回该卡组
3. 候选池有多个卡组时，随机结果必须属于候选池
4. `ResolveDeckContext()` 在本局实际卡组已确定时优先使用它
5. 旧单值配置迁移为数组时结果正确

### 手动回归

至少验证：

1. 主界面勾选多个卡组后，连续两局有概率选择不同卡组
2. 日志能看到每局实际卡组
3. 中控账号保存后重开程序，候选池仍存在
4. 老的 `settings.json` / `accounts.json` 只有 `DeckName` 时仍能正常启动
5. 留牌、发现、学习记录使用的是本局实际卡组名

## 风险与取舍

### 风险

- 主界面和中控同时改卡组配置，容易出现一边支持数组、一边仍按单值写回的状态不一致
- 如果只改自动排队而没改 `ResolveDeckContext()`，学习数据会归到错误卡组
- 如果每次轮询都重新随机，会导致同一局开排前卡组抖动

### 对策

- 明确“每局随机一次”的运行态字段
- UI 配置统一收敛到候选池数组
- `BotService` 作为唯一的本局实际卡组决策点

## 不做的事

- 不做卡组权重随机
- 不做连局去重或保底轮换
- 不改酒馆战棋、竞技场、测试模式的卡组入口
- 不在本次顺带重构整个顶部 UI 布局

## 结论

本次采用“多选候选池 + 每局开排前随机 + 本局运行态固定”的方案：

- 主界面和中控都支持多选卡组
- 每一局匹配前从候选池中随机选择一套卡组
- 运行态、留牌、发现、学习上下文都绑定到本局实际卡组
- 旧的单值配置继续兼容

实现阶段应先补行为测试，再改 `BotService` 的随机入口和上下文绑定，最后接入主界面与中控多选 UI。
