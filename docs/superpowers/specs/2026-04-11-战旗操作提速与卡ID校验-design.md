# 战旗操作提速与 cardId 双重校验 · 设计

**日期**：2026-04-11
**作者**：cc-studi0 与 Claude 协作
**状态**：草案，待复核

## 目标

1. **提速**：战旗模式获得盒子推荐后，一旦游戏内可以买/卖/出，立即执行下一条动作，不再等待固定动画时长。
2. **正确性**：盒子推荐的每一条动作都必须同时匹配 **位置 (position)** 与 **卡牌 ID (cardId)**，避免因 entityId 过期或商店刷新导致买错/卖错/出错牌。

## 非目标

- 不改动非战旗模式（对战/竞技场/酒馆战棋客户端弹窗）的执行路径。
- 不重构 `BgMouseBuy / BgMouseSell / BgMousePlayFromHand` 的鼠标动作协程本身。
- 不引入新的动作队列状态机；现有 `pendingBattlegroundActions` 的 index+list 结构保持不变。

## 现状

战旗主循环位于 `BotMain/BotService.cs:4928-5086`，流程为：

```
while (running && PHASE=RECRUIT):
    recommendation = 读盒子
    actions = recommendation.Actions
    for each action:
        SendActionCommand(pipe, action, 3000)
        SleepOrCancelled(180/220/500)   ← 固定等待动画
```

存在的固定 sleep 点：

| 位置 | 时长 | 触发条件 |
|---|---|---|
| 4924 | 500ms | 非 RECRUIT 阶段空转 |
| 4984 | 180ms | 重复推荐等待刷新 |
| 5049 | 180ms | 同批多条动作之间 |
| 5065 | 220ms | 动作失败退避 |
| 5074/5079 | 150ms | 战旗 choice 处理后 |
| 5082 | 220ms | 单条动作执行后 |
| 5086 | 500ms | 主循环尾部 |

命令格式（`HsBoxRecommendationProvider.ConvertStepToCommand:1259-1418`）：

- `BG_BUY|entityId|position`
- `BG_SELL|entityId`
- `BG_PLAY|handEntityId|targetEntityId|dropPosition`

这些命令只携带 entityId，不携带卡牌稳定标识。执行层 `ActionExecutor.cs:1531-1595` 直接按 entityId 落鼠标。如果盒子推荐里的 entityId 因商店刷新、抽卡、卖牌等原因已经过期，就会买到错误的随从。

## 架构变更

### 新增：`BotMain/BgExecutionGate.cs`

一个轻量执行闸门，单条命令一次调用。职责：

1. **解析** 命令串为 `BgCommandSpec`（含 kind / entityId / position / expectedCardId）。
2. **重解析** 从最新 `stateData` 里按 `(position, cardId)` 查当前真 entityId；cardId 为真源。
3. **一致性判定**：
   - cardId 匹配、entityId 变了 → 重写命令串，标记 `Retargeted`
   - cardId 匹配、position 变了 → 重写 position 段
   - cardId 有多张（商店重复）→ 选距离原 position 最近的一张
   - cardId 找不到 → 返回 `Aborted`，让主循环清空 pending 队列并等刷新
4. **发送** `SendActionCommand(pipe, 修正后命令, 3000)`。
5. **事件驱动探测**：成功后以 20ms 间隔轮询 `stateData`，直到预期变化命中（最多 200ms）；命中则立即返回，超时则 sleep 50ms 兜底（`CompletedWithFallback`）。
6. **失败** 则 `Failed`，主循环按现有 220ms 退避路径继续。

### 修改：`HsBoxRecommendationProvider.cs`

`ConvertStepToCommand`（第 1259 行）生成命令时在末尾追加 cardId 段：

| 动作 | 新格式 |
|---|---|
| Buy | `BG_BUY\|entityId\|position\|cardId` |
| Sell | `BG_SELL\|entityId\|cardId` |
| Play | `BG_PLAY\|handId\|targetId\|dropPosition\|sourceCardId` |

`card.CardId` 已经存在于 `step.GetPrimaryCard()`（第 1337 行日志现已在用），无需改 payload 解析。

### 修改：`BotMain/BotService.cs` 主循环

第 4959-5086 段内的 `SendActionCommand + 多个 SleepOrCancelled` 替换为：

```csharp
var gateResult = _bgGate.Execute(nextAction, recommendation, pipe);
switch (gateResult.Outcome)
{
    case GateOutcome.Completed:
    case GateOutcome.CompletedWithFallback:
    case GateOutcome.Retargeted:
        // 推进 pendingBattlegroundActionIndex，保留现有队列逻辑
        break;
    case GateOutcome.Aborted:
        pendingBattlegroundActions.Clear();
        pendingBattlegroundRecommendationKey = string.Empty;
        pendingBattlegroundActionIndex = 0;
        SleepOrCancelled(180);
        continue;
    case GateOutcome.Failed:
        SleepOrCancelled(220);
        continue;
}
```

尾部 5086 行的 500ms 改为 50ms（PHASE=RECRUIT 空转时保留响应速度）。4924 行的非 RECRUIT 500ms 保持不变（非战旗期不必激进）。

### 修改：`HearthstonePayload/ActionExecutor.cs`

`BG_BUY / BG_SELL / BG_PLAY` 的 case 分支（1531-1595）扩展解析，新增可选 cardId 段透传给 `BgMouseBuy/Sell/Play`。执行层在"即将落鼠标"前（`GetEntityScreenPos` 调用之前）做最后一次 cardId 校验：如果当前 entity 的 cardId 与期望不符，直接返回 `FAIL:bg_card_mismatch`，不消耗动作，不落鼠标。

这一层的校验是闸门的双保险——闸门查的是 `stateData`，但从闸门返回到鼠标落点之间仍有毫秒级窗口让游戏状态变化。

## 数据结构

```csharp
internal enum BgCommandKind { Buy, Sell, Play, Other }

internal readonly struct BgCommandSpec
{
    public BgCommandKind Kind;
    public int EntityId;            // 盒子推荐里的原始 id（可能过期）
    public int Position;            // 1-based zone position
    public int TargetEntityId;      // 仅 Play 使用
    public string ExpectedCardId;   // 为空表示老格式，跳过 cardId 校验
    public string Raw;
}

internal enum GateOutcome
{
    Completed,
    CompletedWithFallback,
    Retargeted,
    Aborted,
    Failed,
    Skipped
}

internal readonly struct GateResult
{
    public GateOutcome Outcome;
    public string ExecutedCommand;  // 可能与原命令不同（Retargeted）
    public string Detail;           // 诊断：如 dup_nearest / probe_timeout / card_missing
}
```

## 期望状态变化谓词

| 动作 | 谓词（满足任一即视为完成） |
|---|---|
| Buy | shopMap 里原 `(pos, cardId)` 条目消失；**或** handMap 尺寸 +1 |
| Sell | boardMap 里原 entityId 消失；**或** board 尺寸 -1 |
| Play | handMap 里原 `(pos, cardId)` 条目消失；**或** board 尺寸 +1 |
| Other（TAVERN_UP/REROLL/FREEZE/END_TURN/HERO_POWER/MOVE 等） | `stateData.GetHashCode()` 与调用前不同 |

`stateData` 是扁平字符串，hash 用 `string.GetHashCode()` 足够——任何 tag 变化都会让它不同。

## 失败与回退

| 场景 | 闸门输出 | 主循环后效 |
|---|---|---|
| cardId 匹配 + 需要修 entityId 或 position | `Retargeted` | 正常推进队列 |
| cardId 同名多张 | `Retargeted`，选距离原 pos 最近的那张，`Detail=dup_nearest` | 正常推进 |
| cardId 找不到 | `Aborted` | 清空 pending 队列，sleep 180ms 等下次刷新 |
| 命令缺 cardId 段（老格式/回放） | 跳过校验，按旧逻辑发命令 | 等同当前行为 |
| `SendActionCommand` 返回 FAIL | `Failed` | 保留 pending 队列，现有 220ms 退避 + stale 检测 |
| 执行层落鼠标前 cardId 不符 | 执行层返回 `FAIL:bg_card_mismatch` → 闸门判 `Failed` | 同上 |
| 探测 200ms 内无变化 | `CompletedWithFallback`，已 sleep 50ms | 推进队列，但复用现有 stale 检测：连续 3 次同 entityKey 触发清空 |

与现有 `staleActionEntityKey / staleActionCount / staleActionFirstUtc`（4609-4614, 5007-5040）完全兼容——`Retargeted` 会让 entityKey 跟着新 id 变化自然绕开旧 stale，`Aborted` 会清空 key，`CompletedWithFallback` 复用现有 stale 计数无需新字段。

## 回滚点

闸门是纯追加的类，命令扩展是向后兼容的：

1. 主循环 `_bgGate.Execute(...)` 换回原来的 `SendActionCommand + Sleep` 段即可关闭整套改动。
2. `HsBoxRecommendationProvider` 不追加 cardId 字段只是少一个段，执行层仍能解析。
3. 两个开关独立可回滚。

### 集成验证（上线前手测 —— 用户负责）

闸门单测覆盖不到真实 Hearthstone 进程段：

1. 开启"跟随盒子推荐"
2. 进一场战旗，观察 `[BG]` 日志是否出现 `gate=Retargeted` / `gate=Aborted` 诊断
3. 确认回合结束前"买-卖-出"一连串动作的 wall-clock 时间明显缩短（对比改动前）
4. 肉眼检查无买错/卖错/出错牌

## 指标预期

改动前单轮"买-出"组合的 wall-clock 下限（不含鼠标移动本身）：

```
SendActionCommand(~30ms) + Sleep220 + SendActionCommand(~30ms) + Sleep220 ≈ 500ms
```

改动后（Completed 快路径）：

```
SendActionCommand(~30ms) + Probe 20ms + SendActionCommand(~30ms) + Probe 20ms ≈ 100ms
```

理论节省约 **400ms / 对动作**。一个 5 动作的回合约节省 1 秒，够完成一次额外刷新决策。
