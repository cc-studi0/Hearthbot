# 云控手机优先巡检台设计

## 概述

本次改造目标不是把现有云控做成复杂后台，而是把它收敛成一个**你自己日常盯单和下少量指令的网站**。

核心使用场景已经确认：

- 你有多台机器，每台机器同一时间只挂一个脚本、一个游戏
- 你会频繁用手机查看状态，而不是只在电脑大屏前操作
- 每个账号开打前都要录订单号
- 账号切换时，旧订单必须自动失效并要求重新录入
- 达到目标段位后，需要网页显著展示，并尽量通过多种提醒方式防漏
- 网页里只保留少量高频操作：录订单号、改目标段位、切脚本/卡组、停止/继续运行

因此首页将从“桌面三列看板”调整为“**手机优先巡检台**”，优先解决以下问题：

1. 手机上打开后 3 秒内知道哪台在打、哪台异常、哪台完成、哪台换号了
2. 让录单、改目标、切脚本等关键操作仍然可做，但不把首页做得过重
3. 在尽量复用现有 `HearthBot.Cloud`、`hearthbot-web`、BotMain 云控能力的前提下补齐缺口

## 设计目标

### 目标

- 首页优先服务“巡检”而不是“配置”
- 手机端查看效率高于桌面端视觉对称性
- 正式完成提醒尽量不漏，且不重复轰炸
- 账号切换时自动清掉旧订单，避免订单和账号错绑
- 操作入口少而明确，不引入复杂排单与业务管理

### 非目标

- 不做客户资料、价格、备注等完整订单系统
- 不做拖拽排单、批量调度、复杂权限控制
- 不把首页变成大图表分析页
- 不做严格意义的“暂停/恢复线程级状态”，网页语义按现有能力定义为“停止 / 继续运行”

## 现状复用

当前项目已经具备以下基础能力，可直接复用：

- `Device` 已包含 `OrderNumber`、`TargetRank`、`StartRank`、`StartedAt`、`IsCompleted`、`CompletedAt`、`CompletedRank`
- BotMain 心跳已上报：
  - `CurrentAccount`
  - `CurrentRank`
  - `CurrentDeck`
  - `CurrentProfile`
  - `TargetRank`
  - `CurrentOpponent`
- BotMain 已支持完成上报：`ReportOrderCompleted`
- 后端已支持已完成订单跨日归档
- 后端已有 Server酱告警能力
- 现有网页已支持：
  - 录订单号
  - 停止 Bot
  - 切换卡组
  - 手动完成

当前主要缺口：

- 首页结构仍偏桌面三列看板，不适合手机高频巡检
- “账号切换自动清单”还没有后端硬规则
- “改目标段位”底层已支持，但云端白名单未放开
- 正式完成提醒缺少网页横幅、浏览器通知和统一去重策略
- 目前没有“疑似完成”态来给脚本漏上报时兜底

## 信息架构

首页采用“总览 + 状态分组 + 设备卡片 + 抽屉详情”的层次。

### 1. 顶部总览区

只保留 4 个指标卡，按巡检优先级排序：

- 在线设备数
- 今日对局数
- 今日完成数
- 异常设备数

不在首页顶部放胜率、趋势图、复杂统计，避免手机首屏被次级信息占满。

### 2. 状态分组 Tab

首页主体改为 4 个分组：

- `进行中`
- `待录单`
- `异常`
- `已完成`

这 4 个分组对应你的真实关注顺序，而不是旧版的固定三列：

- `进行中`：已经有有效订单号、未正式完成、设备当前可工作
- `待录单`：无订单号，或因账号切换导致旧单失效
- `异常`：离线、心跳超时、命令失败、状态异常等需要优先处理的设备
- `已完成`：脚本已正式上报完成的订单，当天保留展示，隔天归档

### 3. 设备状态卡片

首页每台设备显示为一张纵向信息卡，首屏只展示高优先级字段：

- 设备名
- 当前账号
- 订单号或“待录单”警示
- 当前段位 → 目标段位
- 当前状态（对局中 / 运行中 / 切换中 / 空闲 / 离线）
- 当日战绩或本次订单战绩

卡片视觉优先级：

- `待录单` 用明显警示色
- `异常` 用错误色和更强提示
- `已完成` 用完成色，并带“已提醒”标识
- `疑似完成` 用单独状态样式，和正式完成区分

### 4. 二级详情抽屉

点击卡片打开详情抽屉，不在首页直接塞入过多控制区。

详情抽屉包含：

- 机器与账号基本信息
- 订单信息
- 改目标段位
- 切脚本 / 卡组 / 策略
- 停止 / 继续运行
- 最近对局记录
- 最近一次完成/提醒时间

## 主页交互设计

### 首页主流程

用户打开首页后的典型流程：

1. 先看顶部 4 个总览数字
2. 默认停在 `进行中` 分组
3. 快速扫卡片，找到正在打的机器和需处理的机器
4. 若发现 `待录单` 或 `异常`，直接点卡片进入详情处理
5. 若有新完成订单，顶部完成横幅优先展示，并能一键定位到对应卡片

### 待录单卡片

待录单卡片用于两种情况：

- 新机器上线但尚未录单
- 已有订单的机器发生账号切换，旧订单被自动清空

卡片必须直接显示：

- 当前账号
- 设备名
- “需录订单号”或“账号已变化，需重新录单”

可在卡片内直接录入订单号，减少多一步进入详情的成本。

### 进行中卡片

进行中卡片重点展示：

- 当前订单号
- 当前账号
- 当前段位 / 目标段位
- 进度条
- 当前运行状态
- 当前脚本/卡组
- 本次订单战绩

若正在对局中，附带展示：

- 对手职业
- 当前卡组/脚本

### 异常卡片

异常分组优先包含以下状态：

- 心跳超时导致离线
- 命令失败
- 刚换号但尚未重新录单
- 其他无法归入正常进行中的状态

异常卡片要比普通卡片更明显，并展示明确原因。

### 已完成卡片

正式完成后，卡片移动到 `已完成` 分组，展示：

- 订单号
- 账号
- 起始段位 → 完成段位
- 本次战绩
- 完成时间
- 已提醒状态

已完成卡片保留到次日归档逻辑触发。

## 订单生命周期

### 生命周期定义

订单在系统中的流转如下：

```text
设备在线
  -> 待录单
    -> 录入订单号
      -> 进行中
        -> 疑似完成（网页兜底判定，仅展示）
        -> 正式完成（脚本上报）
          -> 已完成
            -> 次日归档
```

### 换号自动清单规则

这是本次改造中的硬规则。

当设备存在有效订单，且心跳中的 `CurrentAccount` 与该订单绑定时记录的账号不一致时，系统立即执行：

- 清空 `OrderNumber`
- 清空 `StartRank`
- 清空 `StartedAt`
- 清空旧完成状态
- 将卡片转入 `待录单`
- 给出“账号已变化，需重新录单”的显式提示

为防止心跳抖动导致误判，账号名临时空白时不应立刻清单，应有短暂缓冲策略。

## 完成判定与提醒策略

### 1. 正式完成

正式完成只以 BotMain 上报 `ReportOrderCompleted` 为主信号。

收到正式完成后，后端执行：

- 将设备标记为 `IsCompleted = true`
- 记录 `CompletedAt`
- 记录 `CompletedRank`
- 推送 `DeviceUpdated`
- 触发完成横幅事件
- 触发浏览器通知和提示音
- 发送微信提醒（Server酱为首选）

同一订单只发送一次正式完成提醒。

### 2. 疑似完成

当网页或后端聚合层发现：

- `CurrentRank >= TargetRank`
- 但尚未收到正式完成上报

则卡片进入 `疑似完成` 状态，仅在网页展示明显提示：

- “已达到目标段位，等待脚本确认”

疑似完成不触发微信推送，只触发站内提示，避免误报。

### 3. 提醒渠道

正式完成的提醒链路包含：

- 首页顶部完成横幅
- 浏览器桌面通知
- 浏览器提示音
- 微信推送（Server酱）

浏览器通知需要提供明确的权限状态提示和开关。

## 网页保留的操作

遵循“少量但高频”的原则，仅保留以下操作：

### 1. 录入/重录订单号

- `待录单` 卡片可直接输入
- 详情页也可编辑

### 2. 改目标段位

- 放在详情抽屉中
- 不占首页主按钮位
- 底层通过现有 `ChangeTarget` 能力下发

### 3. 切脚本 / 卡组 / 策略

详情页提供：

- 卡组选择
- 策略选择
- 应用配置按钮

若当前版本只先接通卡组与目标段位，则策略切换可按现有 `ChangeProfile` 能力补齐。

### 4. 停止 / 继续运行

按钮文案定义为：

- `停止`
- `继续运行`

不使用“暂停 / 恢复”字样，避免与现有底层能力不一致。

## 前端方案

### 主要页面

重点改造 `hearthbot-web/src/views/Dashboard.vue`。

建议拆分或重构组件：

- `DashboardHeaderStats`
- `CompletionBanner`
- `DeviceOverviewTabs`
- `DeviceStatusCard`
- `QuickActionSheet`
- `DeviceDetailDrawer`

### 交互要求

- 默认进入 `进行中`
- 手机端优先纵向滚动
- 桌面端可使用更宽的双列或多列布局，但不改变首页信息优先级
- 完成横幅始终位于总览区下方
- 异常和待录单优先显示明确文案，不依赖用户自己推断

### 视觉原则

- 不再以“完整三列看板”作为首页主结构
- 卡片信息密度适中，首屏只放决策信息
- 操作按钮收进详情层，避免首页失焦
- 已完成、异常、待录单的视觉区分必须明显

## 后端方案

### 数据层调整

现有 `Device` 字段大部分可复用，新增或补充建议：

- 订单绑定账号快照字段
  - 用于检测换号自动清单
- 疑似完成状态或可推导字段
  - 若不新增字段，也可由聚合层动态计算
- 命令执行状态补充返回
  - 便于前端显示“已排队 / 已执行 / 失败”

### 接口与命令调整

需要补齐：

1. `CommandController` 白名单开放 `ChangeTarget`
2. 若前端需要统一修改卡组+策略，可补齐复合配置下发
3. 设备详情接口返回更适合前端直接消费的聚合信息
4. 完成事件中区分“正式完成”和“疑似完成”

### 实时推送

继续沿用 SignalR：

- `DeviceUpdated`
- `DeviceOffline`
- `CommandStatusChanged`
- `NewGameRecord`

并新增或明确：

- `OrderCompleted`
- `OrderCompletionSuspected`

若不新增专门事件，也应保证 `DeviceUpdated` 携带足够前端状态。

## 异常与边界处理

### 设备离线

- 进入异常分组
- 不直接消失
- 保留最后账号、最后段位、最后心跳时间

### 账号名短暂为空

- 不立刻判定换号
- 需要短暂缓冲时间或连续多次心跳确认

### 目标段位为空

- 允许展示
- 不做完成兜底判定

### 重复完成上报

- 只触发一次正式提醒
- 保持幂等

### 指令下发时设备不在线

- 页面明确提示“已排队，待设备上线执行”或“无法发送”
- 避免用户误以为已经立即生效

## 测试策略

本次实现优先覆盖以下关键逻辑：

### 后端

- 换号自动清单
- 正式完成与疑似完成判定
- 完成提醒去重
- `ChangeTarget` 命令白名单放开后的发送与执行
- 归档逻辑不误清未完成订单

### 前端

- 首页分组逻辑
- 卡片状态展示逻辑
- 完成横幅展示与消失逻辑
- 浏览器通知权限与降级处理
- 手机布局下关键信息不溢出、不遮挡

## 实施范围

建议本次实现控制在以下范围内：

1. Dashboard 重构为手机优先巡检台
2. 换号自动清单
3. 正式完成 + 疑似完成双层状态
4. 完成横幅 + 浏览器通知 + Server酱提醒
5. 改目标段位入口接通
6. 切卡组 / 切策略 / 停止 / 继续运行入口收敛到详情抽屉

以下内容延后：

- 更复杂的订单管理
- 批量调度
- 图表首页化
- 深度运营统计

## 总结

本次设计的本质不是“美化云控页面”，而是把云控重新对齐到你的真实使用方式：

- 首页先解决巡检效率
- 订单状态必须与账号绑定
- 完成提醒宁可多一道兜底，也不能漏
- 操作入口尽量少，但关键操作必须能在网页完成

最终产物应是一个你可以长期自己使用的轻量巡检台，而不是一个功能看起来很多、实际手机上不顺手的后台页面。
