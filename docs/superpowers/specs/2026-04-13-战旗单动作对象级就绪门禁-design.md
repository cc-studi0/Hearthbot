# 战旗单动作对象级就绪门禁 · 设计

**日期**：2026-04-13  
**作者**：Codex 与用户协作  
**状态**：待评审

## 目标

1. 战旗招募阶段不再依赖固定动画等待，只要下一条动作对应对象真正可交互，就立刻执行。
2. 覆盖 `买 / 卖 / 出牌 / 换位 / 刷新 / 升本 / 英雄技能 / 目标选择` 全链路。
3. 保留现有 `cardId` 真源校验，避免刷新、三连、时空扭曲或实体重建后误点错误对象。
4. 失败时优先短等待和安全回退，避免“抢点失败后过几秒再试”的明显浪费。

## 非目标

1. 不重写现有鼠标拖拽协程主体，仍复用 `BgMouseBuy / BgMouseSell / BgMousePlay / BgMouseMove / MouseHeroPower / OPTION` 等执行路径。
2. 不把战旗改造成纯网络包执行模式。
3. 不调整构筑、竞技场、留牌、发现等非战旗主流程的 ready 机制。

## 问题背景

当前仓库已经有两层保护：

1. `BotMain/BgExecutionGate.cs`
按 `cardId + position` 把盒子动作重定向到当前真实实体，并在动作后探测状态变化。

2. `WAIT_READY`
在 `HearthstonePayload/ActionExecutor.cs` 中通过 `IsResponsePacketBlocked / PermitDecisionMakingInput / PowerProcessor / ZoneMgr / 手牌布局` 等全局信号判断“游戏是否 ready”。

这两层已经能解决大量错误实体和全局阻塞问题，但仍存在一个关键缺口：

- 战旗刷新商店、升本、部分出牌或特效完成后，`stateData` 往往已经更新；
- `BgExecutionGate` 会因此认为“下一条可以继续”；
- 但下一条要操作的**具体对象**仍在出场 tween、actor 初始化或 stand-in 不可交互阶段；
- 结果就是脚本抢先操作失败，随后重试又能成功。

根因不是“全局还忙”，而是“下一条动作对应对象尚未进入真正可交互状态”。

## 方案选择

### 方案 A：单动作对象级就绪门禁（推荐）

在现有 `BgExecutionGate` 前后保留不变的前提下，新增战旗专用 ready 探测，按具体命令检查源对象和目标对象是否真的可交互。

优点：

- 最贴合“能买就立刻买”的目标；
- 能删掉 `REROLL / TAVERN_UP` 之后的固定 300ms；
- 对问题根因命中最直接。

缺点：

- 需要补一层战旗专用对象探测逻辑；
- 测试面比简单调 sleep 更大。

### 方案 B：继续扩充全局 WAIT_READY

继续向 `WAIT_READY` 添加更多战旗相关的全局忙碌信号。

优点：

- 改动面较小；
- 复用现有接口。

缺点：

- 仍然无法精确回答“下一张牌是否已可交互”；
- 容易继续多等或误判。

### 方案 C：尽量切到内部 API / 网络提交流程

尽可能减少鼠标拖拽，更多依赖内部点击或网络响应。

优点：

- 理论速度最快。

缺点：

- 对版本更新更脆弱；
- 逆向面更大；
- 当前问题并不需要这么激进。

## 决策

采用 **方案 A：单动作对象级就绪门禁**。

保留：

- `BgExecutionGate` 的 `cardId` 重定向与动作后状态探测；
- `WAIT_READY` 作为全局兜底；
- 现有各类鼠标执行协程。

新增：

- 战旗专用 `WAIT_BG_ACTION_READY` / `WAIT_BG_ACTION_READY_DETAIL`；
- 面向具体命令的“源对象 ready + 目标对象 ready + 必要全局硬阻塞”判定。

## 架构设计

### 1. 主循环职责拆分

战旗招募阶段每条动作改为以下顺序：

1. `BgExecutionGate.Resolve`
先按最新 `stateData` 用 `cardId` 找当前真实实体，必要时重写命令。

2. `WAIT_BG_ACTION_READY`
把重写后的命令传入 payload，在游戏进程内只检查这条命令对应的对象是否 ready。

3. `SendActionCommand`
对象 ready 后立即执行鼠标或内部点击动作。

4. `BgExecutionGate` 动作后探测
继续使用当前的状态变化探测确认动作已生效。

### 2. 分层原则

#### 全局 ready 只负责“硬阻塞”

保留这些会导致点击必失败的条件：

- `IsResponsePacketBlocked`
- `PermitDecisionMakingInput == false`
- `IsBlockingPowerProcessor`
- `ZoneMgr.HasActiveServerChange`

这些信号说明游戏状态机还没到可接受操作的阶段，必须阻塞。

#### 对象级 ready 负责“下一条动作能不能立刻做”

只检查当前命令要碰的对象：

- 商店牌
- 手牌牌
- 场上随从
- 英雄技能按钮
- 目标实体
- 发现/抉择目标

如果对象本身已经 ready，就不再因为无关动画或其他区域变化而继续等待。

## 命令级就绪判定

### BG_BUY

源对象：下一条要买的商店牌。

判定条件：

1. 通过 `entityId` 或 `cardId + position` 能解析到当前商店卡对象；
2. `IsActorReady == true`；
3. 能取到屏幕坐标；
4. 若可读到 tween 状态，则要求没有 active tween；
5. 若可读到交互态，则要求对象 `interactive/enabled`。

满足后立即购买。

### BG_PLAY

源对象：手牌中的下一张牌。  
目标对象：若为指向性出牌，则额外检查目标实体。

判定条件：

1. 手牌对象存在；
2. `IsActorReady == true`；
3. `CardStandInIsInteractive == true`；
4. 手牌区域不处于 `hand_layout_updating / hand_layout_dirty`；
5. 若可读到 tween，则没有 active tween；
6. 若存在目标，则目标对象坐标可解且无 active tween。

### BG_SELL

源对象：场上的下一只随从。

判定条件：

1. 场上对象存在；
2. 屏幕坐标可取；
3. 若可读到 `IsActorReady`，则要求 ready；
4. 若可读到 tween，则要求没有 active tween。

### BG_MOVE

源对象：待换位的场上随从。

判定条件与 `BG_SELL` 相同，但允许附加一个更保守条件：

- 若源对象刚经历生成、磁力、三连或前一次换位导致位置变化，需等待其自身 tween 清空后再拖动。

### BG_REROLL / BG_TAVERN_UP / BG_FREEZE

这些动作本身只检查按钮对象可点击：

1. 按钮对象 active；
2. 按钮 enabled；
3. 若存在 tween，则没有 active tween；
4. 输入允许。

执行后**不再固定 sleep 300ms**。  
后续是否继续，交给“下一条动作对象 ready”决定。

### BG_HERO_POWER

源对象：英雄技能按钮 / 对应实体。  
目标对象：若技能需要目标，则再检查目标。

判定条件：

1. 英雄技能对象存在且可见；
2. 未耗尽，且输入允许；
3. 若可读到 ready/tween/active/enabled，要求均处于可交互状态；
4. 若存在目标，则目标对象坐标可解且无 tween。

### OPTION / 目标选择

复用现有 choice 体系，但触发条件改为：

1. 当前明确进入 `TARGET / OPTION_TARGET / DISCOVER / CHOICE` 对应模式；
2. 目标对象或选项对象坐标可用；
3. 对象无 active tween；
4. 原有 choice snapshot 判断 ready。

## 需要新增的 payload 能力

建议在 `HearthstonePayload/ActionExecutor.cs` 内新增以下能力：

1. `DescribeBattlegroundActionReady(string rawCommand)`
返回类似 `READY` / `BUSY:reason=source_tween_active` 的详情字符串。

2. `IsBattlegroundActionReady(string rawCommand)`
返回布尔 ready。

3. 对象探测 helper：
- `TryResolveBattlegroundShopCardObject`
- `TryResolveBattlegroundBoardCardObject`
- `TryResolveBattlegroundHeroPowerObject`
- `IsBattlegroundShopCardReady`
- `IsBattlegroundBoardCardReady`
- `IsBattlegroundActionReadyCore`

可直接复用现有 helper：

- `TryReadCardActorReady`
- `TryReadCardStandInIsInteractive`
- `TryReadCardHasActiveTweens`
- `TryReadGameObjectActive`
- `TryReadEnabledState`
- `TryResolvePlayTargetScreenPos`

## BotService 改动

### 1. 新增战旗对象 ready 等待

在 `BotMain/BotService.cs` 中新增类似方法：

- `WaitForBattlegroundActionReady(PipeServer pipe, string action, ...)`
- `TryGetBattlegroundActionReadyDiagnostic(...)`

轮询策略：

- 第一阶段：20ms 间隔，最多 120 到 180ms，只盯对象 ready；
- 第二阶段：若仍未 ready，再读一次 `WAIT_BG_ACTION_READY_DETAIL` 和必要的全局 ready 详情，决定是否短暂兜底；
- 仍未 ready 时返回 timeout，让主循环走短退避。

### 2. 招募主循环接入点

当前流程中的：

- 动作前 `WaitForGameReady(...)`
- 动作后 `REROLL/TAVERN_UP` 的固定 `SleepOrCancelled(300)`

都要调整。

新的优先级：

1. 先重定向命令；
2. 再等待该命令的对象 ready；
3. 最后发送动作命令；
4. 动作后继续使用 `BgExecutionGateRunner` 探测状态变化。

### 3. 固定等待调整

删除：

- `BG_REROLL` 后固定 300ms
- `BG_TAVERN_UP` 后固定 300ms

保留：

- `Failed` 后的短退避；
- `Aborted` 后等待盒子刷新；
- 招募阶段循环尾部的 50ms 空转。

## 失败与回退

### 1. cardId 未命中

沿用当前 `Aborted` 语义：

- 清空 pending 队列；
- 等推荐刷新；
- 不冒险盲点。

### 2. 对象级 ready 超时

返回 `not_ready_timeout`，主循环只做短退避，不把它误认为全局卡死。

### 3. 执行前对象瞬时变化

继续保留 `ActionExecutor` 中的 `bg_card_mismatch` 双保险，不落鼠标。

### 4. 无法读取对象级信号

如果个别版本下读不到 `IsActorReady / interactive / tween`：

- 退回到“坐标存在 + 无全局硬阻塞”；
- 不让单一反射失败导致整套系统不可用。

### 5. 版本兼容策略

对象级 ready 的每个子判断都应是“尽量读，读不到就降级”，避免版本更新后整个流程全量失败。

## 日志设计

新增高价值诊断原因，便于快速定位：

- `source_missing`
- `source_actor_not_ready`
- `source_tween_active`
- `source_not_interactive`
- `target_missing`
- `target_tween_active`
- `target_pos_not_found`
- `button_disabled`
- `global_blocked:response_packet_blocked`
- `global_blocked:input_denied`
- `global_blocked:blocking_power_processor`
- `global_blocked:zone_active_server_change`
- `not_ready_timeout`

日志目标是让一次失败能够直接区分：

- 是对象还没准备好；
- 还是全局状态机没准备好；
- 还是实体已经漂移。

## 测试策略

### 单元测试

扩展 `BotCore.Tests/BgExecutionGateTests.cs`，并新增对象级 ready 相关测试：

1. `WAIT_BG_ACTION_READY` 对不同命令类型的解析；
2. `BG_REROLL / BG_TAVERN_UP` 后不再依赖固定 sleep；
3. 源对象 tween 未结束时返回 busy；
4. 源对象 interactive 后立即返回 ready；
5. 目标存在但目标 tween 未结束时，指向性动作保持 busy；
6. 对象读取失败时回退到降级逻辑。

### 手动验证

重点验证以下真实场景：

1. 刷新后第一张推荐购买牌，动画结束即刻购买；
2. 升本后第一张推荐购买牌，不再固定等 300ms；
3. 买牌后立刻卖或立刻出牌的连招是否明显提速；
4. 英雄技能与目标选择是否会抢点失败；
5. 时空扭曲、三连、磁力、商店法术等实体重建频繁场景下是否仍稳定。

## 回滚策略

回滚应尽量简单：

1. `BotService` 恢复使用现有 `WaitForGameReady + 固定等待`；
2. 不调用新的 `WAIT_BG_ACTION_READY`；
3. `BgExecutionGate` 与 `cardId` 双保险继续保留。

这意味着对象级 ready 是一层可拆卸优化，不影响现有主框架。

## 风险

1. 某些商店对象可能没有统一的 `IsActorReady / interactive` 反射接口，需要多路径兼容。
2. 不同版本或不同战旗特殊物件可能复用不同 card actor 类型。
3. 过于保守会重新退化为“还是在等”；过于激进则会回到抢点失败，因此必须用日志和测试校准。

## 结论

本次优化的关键不是继续缩短固定 sleep，而是把“能不能做下一步”的判断从**全局忙碌**切换到**下一条动作对应对象是否已真正可交互**。

只要这层对象级 ready 成立：

- 刷新后能买就立刻买；
- 卖和出牌能做就立刻做；
- 不会因为无关动画或整块区域未完全稳定而白等；
- 又能保留当前 `cardId` 双保险，避免误点。
